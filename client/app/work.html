Polestar

Polestar space solutions want to provide customers with additional layers to their existing weather maps. The new weather+ service will display maps with ocean wave and current models.

The existing layout is based on a list of select boxes contained in an accordion layout container. Polestar chose to add two more elements to the accordion.

The design looked like this.

This approach presents a number of usability issues:

  1. The user cannot see at a glance which layers are selected, as they are either hidden by a collapsed state of the container element or because the accordion element overflows in its container. This is a breach of the “visibility of system status” heuristic criterion. It increases cognitive load and requires additional actions from the user.
  2. The first layer is selected by clicking a radio button, styled as a toggle switch. This is neither consistent nor intuitive. It makes the user think: What happens if I push that button? Is it some sort of filter? Why is different from the other layer selections?
  3. Similarly, the layers from the currents group are selected by pressing radio buttons. This violates the “consistency and standards” heuristic criterion, and the “don’t make me think” principle.

In a first attempt to address these problems, I proposed to add the selected layers above the accordion element, in the form of clickable pills, and remove the selected layers from the layers list. The accordion can only have one element open at the time, which eliminates the need to scroll. All layers are also consistently selectable by a select box.
This is the second version.

This design improves on the previous but is still not user-friendly. Even if the system state is visible at all times, it is hard to read. In addition to that, there are multiple sliders, each holding state.
In short, the screen is to complex. A better solution was required.

The reason the selection status is hard to read is because the selected layers are displayed inside their proper group container.
It would be better to move all selected layers to the top, so they become more apparent.
Also, there is no need to have separate sliders for each group, as it would not make sense to overlay maps from different time frames. Given that the accordion headers take up so much vertical space, the accordion is to be replaced by a carousel.

The final design eliminates all issues with previous versions and offers added flexibility to navigate through the layer groups (choice between group icon buttons and previous/next buttons).
It also allows more groups to be added later on without negatively impacting usability. The reclaimed vertical space can be used to add a filter field.

VRT

VRT (Belgian radio and television) wants to optimize their news editing and publication process. Some manual tasks (like incoming news classification, summarization and localization) need to be automated, and the workflow must be centralized.

Currently, the different channels (radio, television, internet) use different applications and workflows to accomplish the same tasks. Some workflows require substantial manual input.

The aim is to integrate these disparate workflows into a single application to be used across all newsrooms, for editing, and subsequent publishing to the different channels.

VRT is a state owned company, and is heavily politicised. As a consequence, there is a high resistance to change. It is important to devise a strategy to reduce friction as the new application is being introduced.

Strategy

Given these constraints the following strategic pillars suggested:

  • Use data driven design as a way to drive acceptance.

    Communicate this viewpoint clearly throughout the organisation. The term ‘data driven’ conveys impartiality and thoughtfulness.

  • Starting from a minimal viable product, use an iterative process with strong feedback loop to push new features.

    Use focus groups to centralise communication. Avoid exchanges to turn into discussion groups. Remind of the data driven design if needed. Provide a prominent feedback button in the new application.

  • Contextual, just-in-time onboarding.

    Reduce friction for new users by providing contextual help.

    Maximize empty states (no content yet). Use it as an opportunity to start demonstrating the value of the new tool.

    Detect which new product features remain underused (on a user profile basis) and remind of them once in awhile. Provide a "do not remind me" option.

    Show in which alternative ways an action can be accomplished. e.g. drag and drop vs context menu

  • Keep the look and feel similar to the old application
  • Identify early adopters and work with them to exert peer pressure

Process

  • Using participant observation and interviews, document the way people go through their everyday work. For each participant create cards to describe their specific Information Architecture. Identify common patterns to select the cards to be used for defining the new IA.

  • Define the personas.
    Share the personas with the focus groups, incorporate feedback.

  • Use the information obtained during observation/interviews to determine the product's core features, and roughly define future milestones. Draw the first wireframes and prototypes. Use the focus groups to create consensus over the functionality and features to be included in the first version.

  • Develop a pilot application. This stage is the most crucial part of the project. Good feedback and relevant data about productivity gains will ensure buy-in from management. Demonstrate the ability to rapidly iterate, encorporating user feedback.

    Once the pilot version is live, continued user testing to detect user needs and reduce friction combined with usage data analysis help to determine focus points for the next iterations.

Plantzr

In 2015, I lived in Portugal for most of the year. Coming from the northern hemisphere, I am familiar with plants like Hogweed, Dandelion and trees like Oak, Beech and Birch. Most plants encountered in the south of Europe where unknown to me.

Carob tree, Rubber fig, Aloe and Agave varieties, Oleander, Giant cane are among those common plants that I had no name for.

I decided to create a mobile application to document the plants I saw around me, and learn more about them.

Visual: the wireframes

The users insert photographs and a description of plants into the aplication. A quiz section in the app lets users identify plants posted by others.
It is also possible to send a tweet with the app's hashtag asking for the twittersphere to respond.

The animated prototype built with framer.js shows the transition between the overview screen and the search screen.
